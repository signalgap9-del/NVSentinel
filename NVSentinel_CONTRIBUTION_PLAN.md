# NVSentinel 기여 가이드: Health Event 유틸리티 및 오류 처리 개선

## 🎯 기여 목표
본 가이드는 NVSentinel 의 핵심 데이터 흐름인 "Health Event" 처리 로직과 오류 재시도 메커니즘을 분석하고, 기존 PR(#1198) 에서 제안된 방향성을 기반으로 초기 기여자가 안전하게 참여할 수 있는 구체적인 작업을 정의합니다.

### 왜 이 주제를 선택했는가?
1. **핵심 동맥**: 모든 헬스 모니터 (GPU, Syslog 등) 는 이벤트를 생성하고, 플랫폼 커넥터가 이를 처리합니다. 이 흐름을 이해하면 전체 아키텍처를 파악할 수 있습니다.
2. **안전한 진입점**: 비즈니스 로직 (복구, 격리) 을 직접 수정하는 것이 아니라, 유틸리티 함수를 정리하는 것이므로 시스템 장애 위험이 상대적으로 낮습니다.
3. **명확한 개선 포인트**: 중복된 코드 제거, 일관된 오류 메시지, 재시도 로직 표준화 등 할 일이 명확합니다.

---

## 🔍 코드 분석: 현재 상태 vs 목표 상태

### 1. Health Event 유틸리티 (`commons/pkg/eventutil`)
**현재 문제점:**
- 여러 모듈 (GPU Monitor, Syslog Monitor 등) 에서 이벤트 정렬, 시간戳 처리, 상태 메시지 생성 로직이 중복되어 있습니다.
- 이벤트 메시지가 너무 길어지거나 포맷이 일정하지 않아 로그 분석이 어렵습니다.

**개선 방향 (PR #1198 참조):**
- **단일 진실 공급원 (Single Source of Truth)**: 모든 헬스 이벤트 생성 로직을 `health_event.go` 로 통합합니다.
- **표준화된 포맷**: 
  - `SortEvents`: 타임스탬프 기반 정렬 로직 일원화
  - `GenerateStatusReason`: 오류 원인을 인간이 읽기 쉬운 형태로 표준화
  - `TruncateMessage`: 데이터스토어 제한 길이에 맞춰 자동으로 잘라내기 (로그 손실 방지)

### 2. 오류 처리 및 재시도 (`commons/pkg/errutil`)
**현재 문제점:**
- `context.Canceled` 와 같은 일시적 오류와 영구적 오류의 구분이 모호합니다.
- 각 모듈마다 재시도 로직 (Backoff 등) 을 다르게 구현하고 있어 유지보수가 어렵습니다.

**개선 방향:**
- **일시적 오류 감지 강화**: `IsTemporaryError(err)` 함수를 통해 네트워크 끊김, 락 경합 등 재시도가 필요한 오류를 명확히 분류합니다.
- **재시도 로직 이관**: 개별 모듈의 재시도 코드를 제거하고, 공통 `errutil` 에 정의된 표준 재시도 정책을 따르도록 변경합니다.

## 📝 기여 작업 계획 (Step-by-Step)

### 단계 1: 환경 설정 및 테스트 통과 확인
- [ ] 로컬에 NVSentinel 클론 및 의존성 설치
- [ ] `commons` 모듈의 기존 유닛 테스트 실행 (`go test ./commons/pkg/...`)
- [ ] 현재 코드베이스의 린트 (lint) 오류 확인

### 단계 2: Health Event 유틸리티 분석 및 정리
- [ ] `commons/pkg/eventutil/health_event.go` 파일 분석
- [ ] 다른 모듈 (예: `health-monitors/gpu-health-monitor`) 에서 중복 구현된 이벤트 생성 로직 찾기
- [ ] 기존 함수들이 다음을 올바르게 처리하는지 검증:
  - 이벤트 시간순 정렬
  - 오류 메시지의 일관된 포맷팅 (예: `[XID 123] GPU fell off the bus`)
  - 긴 메시지의 안전한 잘라내기 (UTF-8 문자 깨짐 방지)

### 단계 3: 오류 처리 유틸리티 개선
- [ ] `commons/pkg/errutil/errors.go` 파일 분석
- [ ] `IsTemporaryError` 함수가 `context.DeadlineExceeded`, `connection refused` 등을 제대로 감지하는지 확인
- [ ] **중요**: `context.Canceled` 가 일시적 오류로 잘못 분류되지 않도록 로직 점검 (사용자 취소는 재시도 불필요)

### 단계 4: 작은 규모의 리팩토링 PR 작성
- [ ] **범위 제한**: 새로운 기능을 추가하는 대신, 기존 코드를 공통 유틸리티로 교체하는 작업만 수행
  - 예: `gpu-health-monitor` 내의 중복된 이벤트 정렬 코드를 `eventutil.SortEvents` 호출로 변경
- [ ] **테스트 추가**: 변경된 유틸리티 함수에 대한 유닛 테스트 보강 (경계 조건 테스트 포함)
- [ ] **문서화**: 함수 주석 (Godoc) 을 명확히 작성
